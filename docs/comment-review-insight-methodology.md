# 评论数据产品洞察方法论

本文档沉淀自 VR 前 100 免费内购真联机产品评论研究流程，目标是把一次性报告生成经验整理成可迁移、可复用、可审计的方法论。它适用于竞品评论分析、应用商店评论挖掘、游戏需求洞察、产品机会评估和新品方向判断。

## 1. 核心目标

评论研究不是为了复述玩家说了什么，而是为了找到产品价值提升机会。

默认产品价值公式：

```text
产品价值 = 新体验 - 旧体验 - 替换成本
```

因此评论分析需要回答三类问题：

1. 玩家已经被什么体验吸引。
2. 玩家在哪些地方持续不满或流失。
3. 新产品或新功能怎样提供足够强的新体验，抵消竞品已有好友关系、资产沉淀、操作习惯、内容记忆和品牌惯性。

## 2. 第一性原理

### 2.1 评论是行为后的压缩信号

评论不是完整用户研究，但它是玩家完成体验后的主动表达。它天然带有偏差：极端满意、极端不满、被活动激励、低龄短评、刷评和历史版本遗留都会混在一起。因此不能把评论总数直接等同于真实需求强度。

正确做法是把评论拆成三层：

- 事实层：评分、时间、长度、有帮助数、原文。
- 表征层：优势点、许愿墙、痛点池。
- 推理层：潜在需求、产品机会、替换成本判断。

### 2.2 产品样本价值和评论样本价值必须分开

一个产品满足研究样本要求，不代表它适合深描。

产品样本价值由以下因素共同决定：

- 榜单排名：代表当前活跃和市场验证。
- 商业模式：免费内购真联机优先。
- 联机真实性：公开房、好友房、多人合作/对战，而不是伪联机或单机。
- 玩法归属：是否能进入当前研究类目。
- 评论总量：是否足以支撑归纳。
- 高价值评论供给：是否有足够长评、高赞、近期、低星长评。

评论样本价值由以下因素共同决定：

- 长度：越长越可能包含具体场景、原因和建议。
- 有帮助数：越高越说明共鸣或代表性更强。
- 时间：近期评论更能反映当前版本。
- 评分：低星长评适合挖痛点，高星长评适合挖做对点。
- 信息量：是否包含具体机制、场景、对比、后果、诉求。

## 3. 标准流程

### 3.1 阶段一：样本池校准

先建立候选样本，不直接生成正式报告。

候选样本需要记录：

- 产品名
- 当前榜单排名
- 价格和内购状态
- 联机证据
- 玩法/类目/分支
- 纳入理由
- 边界疑问
- 评论总量

当用户或研究负责人确认类目边界后，再进入评论重读。

### 3.2 阶段二：全量评论扫描

从源表全量读取评论，而不是复用旧报告文本。

推荐字段：

- 产品标识
- 产品名
- 评分
- 评论文本
- 有帮助数
- 评论时间戳
- 当前榜单排名
- 商店评分和评分数

必须保留可审计口径：

- 数据源表
- 扫描产品数
- 扫描评论总数
- 生成时间
- 是否剔除无效时间戳
- 是否排除弱证据产品

### 3.3 阶段三：评论价值评分

每条评论应先计算评论价值分，再进入主题归纳。

默认高价值评论判定：

```text
高价值评论 =
  长度 >= 240
  或 有帮助数 >= 20
  或 低星且长度 >= 120
  或 近期且长度 >= 120 且有帮助数 >= 5
```

默认评论价值分：

```text
value =
  min(length, 800) / 35
  + min(helpful, 300) * 2.4
  + length_bonus
  + helpful_bonus
  + recent_bonus
  + low_star_dense_bonus
```

默认加分：

- 长度 >= 240：+18
- 长度 >= 120：+8
- 有帮助数 >= 20：+28
- 有帮助数 >= 5：+10
- 近期评论：+18
- 低星且长度 >= 120：+24

这些阈值不是固定真理，应按数据规模调整。关键原则是：不要让大量短评淹没少量高信息密度评论。

### 3.4 阶段四：产品证据等级

产品必须按评论证据质量分层。

默认弱证据判定：

```text
弱证据 =
  评论总量 < 500
  或 高价值评论 < 75
  或 240 字以上长评 < 50
```

默认强证据判定：

```text
强证据 =
  评论总量 >= 1000
  且 高价值评论 >= 200
  且 240 字以上长评 >= 100
```

其他为中证据。

处理规则：

- 强证据：可进入正式单品深描。
- 中证据：可保留，但结论要降权。
- 弱证据：不进入正式深描，只作为分支补样或剔除清单保留。

### 3.5 阶段五：主题提取

主题提取不是最终结论，只是证据索引。

建议分三类：

- 优势点：好玩、朋友联机、移动手感、战斗爽感、收集养成、恐怖刺激、用户创作、模拟体验、免费公平。
- 许愿墙：更多地图/模式/内容、更多奖励/经济路径、更多物品/武器/能力、私房/好友、教程、新手引导、交易保护、排位/赛事、创作工具。
- 痛点池：缺陷/卡顿/崩溃、服务器/匹配、毒性社区、误封/审核、外挂/作弊、付费赢/付费墙、资产丢失、太肝、平衡/手感、内容重复。

主题排序应优先使用：

1. 主题评论价值分总和。
2. 高价值评论数量。
3. 全期主题命中数。
4. 近期阶段命中数。
5. 有帮助标记数。

不要只按关键词命中总数排序。

### 3.6 阶段六：逐条阅读高价值评论

全量扫描不等于逐条阅读。

必须对每个正式样本抽取高价值评论池，并逐条阅读或语义归纳：

- 价值评论前若干条
- 近期低星长评
- 全期高赞长评
- 近期高赞长评
- 主题内代表样本

阅读时记录：

- 玩家直接说了什么
- 具体场景是什么
- 情绪强度和后果是什么
- 是否仍是近期问题
- 是显性需求、潜在需求、还是历史遗留
- 能否映射到产品机制或商业化策略

报告引用原文时可以截断，但内部分析应基于完整评论。

### 3.7 阶段七：生命周期切分

按评论生命周期三段：

- 前期：首评到生命周期 1/3。
- 中期：生命周期 1/3 到 2/3。
- 近期：生命周期 2/3 到末评。

同时单独标注当前自然时间近期，例如 2026 年评论。

用途：

- 判断问题是否历史遗留。
- 判断更新后是否改善。
- 判断新品窗口是否仍存在。
- 避免把早期缺陷误当当前痛点。

无效时间戳需要排除或标注，不能进入时间线推理。

### 3.8 阶段八：单品分析框架

每个正式样本按三层输出。

#### 01 总体概述

- 产品轮廓
- 目标人群
- 评分口碑
- 证据等级
- 评论总量
- 高价值评论数
- 长评数
- 中位长度
- 低星比例
- 有帮助标记数
- 近期评论占比
- 生命周期三段

#### 02 特点分析

表征提取：

- 优势点
- 许愿墙
- 痛点池

深入挖掘：

- 定位复盘
- 痛点归因
- 需求洞察
- 优势放大

每条主题结论应包含：

- 量化依据
- 直接需求
- 潜在需求
- 产品处理方向
- 代表性原文

#### 03 预测推理

- 玩法推演
- 变现推演
- 新品可能

推理必须回到：

```text
产品价值 = 新体验 - 旧体验 - 替换成本
```

### 3.9 阶段九：类目洞察

类目洞察不是把单品结论相加，而是找类内共同结构和分支差异。

输出结构：

- 类目概述
- 共同结构
- 分支差异
- 深入挖掘
- 类目推理

重点回答：

- 类目共同需求是什么
- 哪些需求只属于某个分支
- 哪些痛点是结构性问题
- 哪些优势说明市场已经验证
- 新品应该正面竞争、错位切入，还是只做垂直补位

### 3.10 阶段十：报告质量审查

生成后必须做对抗审查：

- 是否使用最新数据源。
- 是否真的从源评论表重扫。
- 是否明确样本池和剔除规则。
- 是否把弱证据产品降级或剔除。
- 是否展示评论价值指标。
- 是否用高价值评论支撑结论。
- 是否避免英文产品描述污染中文报告。
- 是否排除无效时间戳。
- 是否区分历史问题和近期问题。
- 是否有结论超出证据。

## 4. 推荐输出物

标准输出至少包括：

- `data.json`：结构化分析结果。
- `index.html` 或报告页：阅读型报告。
- 样本池清单：纳入/剔除和原因。
- 方法说明：口径、阈值、证据等级。
- 验证结果：评论数、产品数、访问状态、解析状态。

## 5. 可迁移参数

不同领域可以调整：

- 高价值评论长度阈值。
- 有帮助数阈值。
- 近期定义。
- 弱证据剔除阈值。
- 主题词表。
- 类目和分支定义。
- 是否需要人工翻译原文。
- 是否需要抽样人工复核比例。

不要改的原则：

- 先定样本池，再写报告。
- 先评估评论价值，再做主题归纳。
- 产品证据弱就降级，不强行深描。
- 结论必须对应原文、统计和时间线。
- 把用户显性需求、潜在需求、产品机会分开。

## 6. 最小可复用流程

```text
1. 建候选样本池
2. 确认类目和分支
3. 全量读取评论
4. 清洗无效时间戳和空文本
5. 计算评论价值分
6. 计算产品证据等级
7. 剔除弱证据样本
8. 提取优势点/许愿墙/痛点池
9. 逐条阅读高价值评论池
10. 生成单品画像
11. 生成类目洞察
12. 对抗审查并输出报告
```
