# Comment Review Insight Skill

`comment-review-insight` 是一套把“大规模用户评论”转成“产品研究洞察”的 Codex skill 和方法论。

当前版本：`1.0.0`

它适合这些场景：

- 应用商店评论分析
- 游戏竞品评论研究
- Top 榜单样本重聚类
- 用户需求、痛点、许愿和优势提取
- 产品机会、新品方向和替换成本判断
- 已有评论报告的质量审查和反模板化复核

## 核心思想

这不是评论摘要工具。它的目标是通过评论数据找到能指导产品设计和开发的需求线索。

默认思维框架：

```text
重点关注 -> 分类提取 -> 分层处理
```

默认产品价值坐标：

```text
产品价值 = 新体验 - 旧体验 - 替换成本
```

默认证据链：

```text
raw comment -> comment value -> structured insight -> theme -> product judgment -> product action
```

如果一个结论不能从原始评论、评论价值、结构化洞察、主题聚合和产品动作一路追溯，它就只能算弱信号。

## 1.0.0 固化的原则

- 信息量高于信息长度：短但具体的评论，可能比长但空泛的评论更有价值。
- 高赞只是放大器：高赞低信息评论代表情绪共鸣，高赞高信息评论才适合进入产品动作依据。
- 重复文本不重复计权：同一产品内的规范化重复评论只能作为一个证据族。
- 时间线服务于当前有效性：时间分析主要用来判断历史问题是否仍然有效。
- 单品分析必须保留差异：产品专属实体、旧体验参照和循环断点不能被模板吞掉。
- 类目洞察必须比较分支：共同结构和分支差异都要出现，否则容易变成泛泛总结。
- 剔除样本不是垃圾桶：前 500 中评论数高但暂不合格的产品，应作为后续补充池保留。
- 发布版必须可回看：确认后的报告应带版本号和冻结副本，避免后续迭代覆盖唯一里程碑。

## 推荐工作流

1. 建立候选样本池：先列产品、排名、价格、IAP、联机、类目、评论量和纳入/剔除理由。
2. 锁定样本边界：确认哪些产品进入正式样本，哪些进入弱证据剔除或后续补充池。
3. 全量读取评论：从源表、源文件或 API 读取评论，不复用旧报告正文。
4. 计算评论价值：综合信息量、长度、有帮助数、时间、评分和信息密度。
5. 控制重复文本：同一产品内重复评论只作为一个证据族计权。
6. 分层产品证据：强证据深描，中证据降权，弱证据剔除或补样。
7. 分类提取主题：优势点、许愿墙、痛点池分别统计权重和时效。
8. 深读高价值评论：逐条抽取机制、场景、情绪、功能、显性需求和隐性需求。
9. 生成单品研究：产品概览、证据质量、体验模型、价值拆解、信号洞察、机会推演。
10. 生成类目洞察：类目概述、共同结构、分支差异、深入挖掘、类目推理。
11. 对抗审查报告：检查数据源、重复文本、时效、证据越界和模板化表达。
12. 发布版本：生成结构化数据、HTML/Markdown 报告、默认入口和版本冻结副本。

## 评论价值判断

默认高价值评论规则：

```text
high_value =
  information_score >= 45
  OR (length >= 240 AND information_score >= 28)
  OR (helpful >= 20 AND information_score >= 16)
  OR (rating <= 2 AND length >= 120 AND information_score >= 28)
  OR (recent AND length >= 120 AND helpful >= 5 AND information_score >= 28)
```

默认价值分：

```text
value =
  information_score * 1.8
  + min(length, 800) / 70
  + min(helpful, 300) * 2.4
  + length_bonus
  + helpful_bonus
  + recent_bonus
  + low_star_dense_bonus
```

信息量优先看：

- 具体机制：地图、房间、交易、服务器、首领、武器、经济、审核等。
- 场景和因果：什么时候发生、为什么发生、带来什么后果。
- 明确诉求：希望增加什么、修复什么、解释什么。
- 后果强度：丢失资产、封禁、崩溃、无法加入、被骗、退款。
- 对比参照：更新前后、竞品对比、旧体验迁移。
- 资源和数字：价格、币、等级、概率、时长、数量。

## 单品研究结构

单品研究建议固定为六段，但内容必须从评论里长出来。

1. 产品概览：产品轮廓、用户画像、资源偏好、差异定位、直接证据、定位判断。
2. 证据质量：评论规模、唯一文本、重复文本、高价值评论、高信息评论、时间覆盖。
3. 体验模型：核心动作、循环目标、资源系统、社交结构。
4. 价值拆解：情绪价值、功能价值、替换成本、体验断点。
5. 信号洞察：优势信号、许愿信号、痛点信号、差异信号、代表性原文。
6. 机会推演：玩法机会、变现边界、新品方向、风险判断。

硬性反模板规则：

```text
如果把产品名换成同类另一个产品名，句子仍基本成立，就判为模板化，需要重写。
```

## 类目洞察结构

类目洞察不是把单品结论相加，而是找同类产品的共同结构和分支差异。

推荐结构：

1. 类目概述
2. 共同结构
3. 分支差异
4. 深入挖掘
5. 类目推理

每条类目推理至少绑定：

- 一个高权重共同主题
- 一个明确分支差异
- 一个替换成本障碍
- 一个可执行产品方向

## 发布前检查

- 是否使用最新数据源。
- 是否从源评论重扫，而不是复用旧报告正文。
- 是否清楚列出正式样本、剔除样本和补充池。
- 是否展示评论价值和产品证据等级。
- 是否逐条阅读高价值评论或形成结构化 `comment_insight`。
- 是否对重复文本去重计权。
- 是否区分历史问题、继续有效问题和需要复核问题。
- 是否区分情绪价值和功能价值。
- 是否区分核心体验和循环体验。
- 是否通过产品名替换测试和类目名替换测试。
- 是否生成默认入口、结构化数据和版本冻结副本。
- 是否把可复用的新反馈同步回 `SKILL.md` 和方法论文档。
